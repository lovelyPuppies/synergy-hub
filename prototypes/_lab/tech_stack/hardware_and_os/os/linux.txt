  ☑️ (Issue: Error); Ensure NVIDIA GPU is accessible by correctly setting user permissions in Container Environment 📅 2024-10-20 22:35:55

    ## Issue

    NVIDIA GPU devices are not accessible in the container environment.

    ## When

    - When connecting directly using the following command:

      ```bash
      docker run --runtime nvidia -it --rm --network=host jayfalls/l4t-20.04:full-cp311
      ```

      and executing:

      ```bash
      python3 -c 'import torch; print(f"PyTorch version: {torch.__version__}");   print(f"CUDA available:  {torch.cuda.is_available()}");   print(f"cuDNN version:   {torch.backends.cudnn.version()}"); print(torch.__config__.show());'
      ```

      - **CUDA** will be reported as **True**.
      
    - However, when using the **VSCode DevContainer** with a new user defined in the Dockerfile, the same command will return **CUDA: False**.

    ## Inference

    - The device files are owned by the `root` user and the `video` group.
    - Users not in the `video` group cannot access these devices, restricting GPU usage.

    ## Root cause

    - Non-root users need access to `/dev/nvhost-*` or `/dev/nvidia*` devices.
    - If the user is not in the `video` group, CUDA-related frameworks like PyTorch or TensorFlow will report that the GPU is unavailable.

    ## Solution

    1. Add the user to the `video` group:

      ```bash
      sudo usermod -aG video <username>
      ```

    2. Verify the device files and group ownership with:

      ```bash
      ls -l /dev | grep nv
      ```

    3. Example output:

      ```bash
      crw-rw---- 1 root video 246, 0 Oct 20 14:35 /dev/nvhost-ctrl
      crw-rw---- 1 root video 246, 1 Oct 20 14:35 /dev/nvhost-ctrl-gpu
      ```

    4. Log out and back in or restart the shell session to apply the group change:

      ```bash
      su - <username>
      ```

  ⚓📎 Out of memory (OOM) ; https://en.wikipedia.org/wiki/Out_of_memory
    Out of memory (OOM) is an often undesired state of computer operation where no additional memory can be allocated for use by programs or the operating system. Such a system will be unable to load any additional programs, and since many programs may load additional data into memory during execution, these will cease to function correctly. This usually occurs because all available memory, including disk swap space, has been allocated.
    
    # Recovery
      The kernels of operating systems such as Linux will attempt to recover from this type of OOM condition by terminating one or more processes, a mechanism known as 🪱 the OOM Killer

  ☑️ (Issue); Raspberry PI Display shows only a black screen when switching to a custom kernel image. 📅 2025-01-20 13:05:12
    ➡️ Solution
      Before booting with a custom kernel image, ensure that the modules are installed in `/lib/modules/$kernel_release`.
      Or, If the system has already booted without the required modules, 
        - Install the kernel's modules
        - %shlle> sudo depmod -a && sudo reboot

    ❔ When
      - Build Configuration and Build in Host Raspberry Pi kernel source:
        ```fish
        #!/usr/bin/env fish

        make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- bcm2711_defconfig
        
        ## ❗ Set custom CONFIG_LOCALVERSION
        set CONFIG_FILE .config
        set new_local_version "-v8-synergy_hub"

        set TMP_FILE (mktemp)

        awk -v new_local_version=$new_local_version '{
            if ($1 ~ /^CONFIG_LOCALVERSION=/) {
                print "CONFIG_LOCALVERSION=\"" new_local_version "\""
            } else {
                print $0
            }
        }' $CONFIG_FILE > $TMP_FILE

        mv $TMP_FILE $CONFIG_FILE
    

        ## Build kernel and modules
        make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- Image modules -j(nproc)
        ```
      - Install the Image to /boot/firmware/ on the Raspberry Pi.
      - Reboot the Raspberry Pi.
    
    ❔ Symptom
      Boot message:
        ```plaintxt
        Mounting proc-sys-fs-binfmt_misc.mou…trary Executable File Formats File System...
        [FAILED] Failed to mount proc-sys-fs-binfmt_m…bitrary Executable File Formats File System.
        [FAILED] Failed to start systemd-binfmt.service - Set Up Additional Binary Formats.
        [*     ] (1 of 3) Job plymouth-quit-wait.service/start running (1min 28s / no limit)
        [   ***] (2 of 3) Job dev-dri-card0.device/start running (1min 26s / 1min 30s)
        [ ***  ] (3 of 3) Job dev-dri-renderD128.device/start running (1min 23s / 1min 30s)
        ```
      And Raspberry PI Display shows only a black screen 
      
      🚣 But SSH can be connected.

    🤔 Inference

      1. Copy the new kernel image to /boot/firmware, configure `/boot/firmware/config` to boot from it, and reboot
        --> The screen remains black.
      2. Install modules to `/lib/modules/$kernel_release` and reboot
        --> The screen remains black.
      3. Reset `/boot/firmware/config` to use the default kernel and reboot
        --> The screen works.
      4. Configure `/boot/firmware/config` to boot with the custom kernel again
        --> The screen remains black.
      5. Run `sudo depmod -a` and reboot
        --> The screen works. Subsequent boots with this kernel also work without issues.

    ❔ Root Causes
      ❗ The issue arises from the kernel's module loading mechanism.
        1. The kernel relies on the dependency information files (modules.dep, modules.alias, etc.) in /lib/modules/<kernel_release>/ to automatically load the required modules.
        2. The kernel retains the module information from the first boot with the custom kernel until the dependencies are manually updated (e.g., sudo depmod -a).
          This means that if modules are missing during the first boot with the custom kernel, even installing the modules later will not resolve the issue until the dependency information is manually updated.
      
      Additionally, because the system was initially booted without the necessary modules being installed, even installing the modules afterward and rebooting did not resolve the issue immediately.
